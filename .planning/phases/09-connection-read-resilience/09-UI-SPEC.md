## Component Inventory (Phase 9 Scope)

**No new UI components.** This phase modifies backend behavior only. The existing frontend patterns already support all required display behaviors.

### Existing Patterns Reused (No Modification Needed)

#### 1. Disconnect Button (Existing)

| Property | Value |
|----------|-------|
| Element | `<button id="btn-connection">` |
| Connected state | Class `.btn--disconnect`, background `--color-destructive` (#d9534f), label `Disconnect` |
| Behavior change | None in frontend. Backend now aborts reads within 1s (D-01, D-02). UI waits for `state_change:disconnected` WebSocket message (D-03, already implemented). |

**Interaction flow (D-03, unchanged in frontend):**
1. User clicks Disconnect.
2. Browser sends `{ type: "disconnect" }` via WebSocket.
3. Browser waits for `state_change:disconnected` message from backend.
4. On receipt, UI transitions to disconnected state (status dot, form fields, timers).

No optimistic UI update. The backend's abort-then-disconnect pattern (D-01, D-02) ensures the confirmation arrives within ~1 second.

#### 2. Stale Value Indicator (Existing -- Phase 7)

| Property | Value |
|----------|-------|
| Class | `.data-row-h__value--stale` (style.css line 579) |
| Text color | `--color-text-muted` (#888888) |
| Warning icon | `::after` pseudo-element with content `\26A0` (warning triangle), color `--color-destructive` (#d9534f), font-size 12px, margin-left 4px |
| Applied when | `handleRegisterValue` receives `msg.error` (app.js line 1006) |
| Removed when | `handleRegisterValue` receives successful value (app.js line 1010) |

**How D-08/D-09/D-10 map to existing behavior:**

| Decision | Existing Implementation | Change Needed |
|----------|------------------------|---------------|
| D-08: Suppress errors during retries, keep previous value visible | Backend `executeRead` only returns after all attempts exhausted. Frontend never sees intermediate retry errors. | None -- retry suppression is inherently backend-side |
| D-09: After all retries fail, show stale indicator if previous value exists | `handleRegisterValue` adds `.data-row-h__value--stale` class on error, preserving previous text content (app.js line 1006) | None -- already implemented |
| D-09: If no previous value, show em-dash | Element starts with em-dash (`\u2014`) and class `.data-row-h__value--pending`; on error, class changes to `--stale` but text stays as em-dash | None -- already implemented |
| D-10: Successful retries display value normally | `handleRegisterValue` sets `textContent` and removes `--pending` and `--stale` classes on success (app.js line 1009-1010) | None -- already implemented |

#### 3. Status Indicator States (Existing)

| State | Dot Class | Text | Animation |
|-------|-----------|------|-----------|
| Disconnected | `.status-dot--disconnected` | `Disconnected` | none |
| Connecting | `.status-dot--connecting` | `Connecting...` | pulse (1.2s) |
| Connected | `.status-dot--connected` | `Connected` | none |
| Reconnecting | `.status-dot--reconnecting` | `Reconnecting...` | pulse (1.2s) |

**Note on brief reconnecting flash (RESEARCH.md Open Question 1):** During disconnect-while-reading, the backend may emit a brief `StateReconnecting` before `StateDisconnected`. The frontend already handles this transition correctly -- `handleStateEvent` for `reconnecting` cancels refresh timers and disables nav, and the subsequent `disconnected` event completes the transition. The flash is milliseconds at most and functionally harmless. No frontend change needed.

## Interaction States

### Disconnect While Reading (New Behavior, Existing UI)

| State | What User Sees | What Changed from Phase 8 |
|-------|---------------|---------------------------|
| Reading in progress, user clicks Disconnect | Button click registers immediately. Status dot stays at current state. | Nothing visible -- backend now aborts the blocking TCP read via `SetReadDeadline(time.Now())` |
| Backend processes disconnect (~100-500ms) | Brief flash of "Reconnecting..." possible (milliseconds) | New: the abort path may produce a transient reconnecting state |
| Backend confirms disconnected | Status dot turns red, text shows "Disconnected", form fields re-enable, refresh timers clear | Same visual result as Phase 8, but now guaranteed within ~1 second even during reads |

**Timing guarantee:** User clicks Disconnect -> UI shows "Disconnected" within 1 second, even if a 10-second TCP read was in progress. Previously, the UI could block for up to 10 seconds (the read deadline).

### Register Retry (Invisible to User)

| Scenario | What User Sees | Backend Behavior |
|----------|---------------|------------------|
| Register succeeds on first attempt | Value appears normally | No change from Phase 8 |
| Register fails once, succeeds on retry | Value appears normally (D-10) | Backend retries transparently; 500ms+ delay per retry (inter-read delay + reconnect) |
| Register fails all 3 attempts | Previous value shown dimmed with warning triangle (D-09) | Backend tried 3 times (was 2 in Phase 8), then surfaces error |
| Register returns illegal address (0x02) | Previous value shown dimmed with warning triangle | Backend skips retry, returns error immediately (D-06) |

**User-visible impact:** Fewer stale-value warning triangles during normal operation, because 3 retry attempts recover from more transient errors than the previous 2 attempts.
