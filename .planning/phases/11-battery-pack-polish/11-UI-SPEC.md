## Phase 11 UI Changes

### Change 1: Group Display Order (BATT-01, D-03)

**Current order:** Info, Cells, Temps, Status, Balance
**Required order:** Info, Cells, Balance, Temps, Status

This is a backend change (reorder group append calls in `buildPackDataMessage` and schema group ordering). Frontend renders groups in the order received. No CSS or layout changes needed -- the existing sequential rendering loop in `renderPackDetail` (and its streaming replacement) iterates groups in arrival order.

### Change 2: Pack Streaming Architecture (BATT-02, D-01, D-02)

Pack drill-down switches from batch rendering to the per-register streaming pattern used by main sections.

#### 2a. Pack Schema Message

The backend sends a `section_schema` message (type `"section_schema"`) before each pack read cycle. The schema defines groups and their registers so the frontend can build skeleton DOM.

**Pack schema group types and their skeleton rendering:**

| Group | Schema Type | Skeleton Rendering |
|-------|-------------|-------------------|
| Info | `"standard"` | Standard `group-card` with `data-row-h` rows, `--pending` em-dash values |
| Cells | `"cell_grid"` | `group-card` containing cell-summary (values as `--pending` em-dashes) + cell-grid with 4-column grid of `cell-voltage` divs showing `--pending` em-dash values |
| Balance | `"balance"` | `group-card` with single `--pending` placeholder: "Balanced" text dimmed |
| Temps | `"standard"` | Standard `group-card` with `data-row-h` rows, `--pending` em-dash values |
| Status | `"pack_status"` | `fault-card fault-card--clear` skeleton with `--pending` placeholder |

**Schema message structure (new fields on existing section_schema):**

```
{
  type: "section_schema",
  section: "bms",
  pack_context: { input: 1, tower: 1, pack: 3 },
  groups: [
    { name: "Pack Info", type: "standard", layout: "full", registers: ["Pack Voltage", "Pack Current", ...] },
    { name: "Cell Voltages", type: "cell_grid", layout: "full", cell_count: 16 },
    { name: "Balance State", type: "balance", layout: "full", registers: ["balance_bitmap"] },
    { name: "Temperatures", type: "standard", layout: "full", registers: ["Temp 1", "Temp 2", ...] },
    { name: "Pack Status", type: "pack_status", layout: "full", registers: ["alarm", "protection", "fault", ...] }
  ]
}
```

#### 2b. Skeleton Rendering Rules

When the frontend receives a `section_schema` with `pack_context`, it builds the pack drill-down skeleton:

1. Clear `content-body`
2. Render breadcrumb bar (unchanged from current implementation)
3. For each group in order:
   - **`"standard"` type:** Use existing `renderSkeletonCard(group)` -- creates `group-card` with `data-row-h` rows, each value element has `data-row-h__value--pending` class and em-dash content
   - **`"cell_grid"` type:** Use new `renderCellGridSkeleton(group)` -- creates `group-card` with cell-summary (4 items, all showing em-dash) + cell-grid (N cells based on `cell_count`, each showing cell number + em-dash value)
   - **`"balance"` type:** Use new `renderBalanceSkeleton(group)` -- creates `group-card` with pending balance status text
   - **`"pack_status"` type:** Use new `renderPackStatusSkeleton(group)` -- creates `fault-card fault-card--clear` with pending placeholder
4. Restore cached values from `sectionCache` using composite key `bms:pack:{input}:{tower}:{pack}` (Phase 10 cache integration)
5. If cached values restored, add `content__body--refreshing` class to dim them

#### 2c. Per-Register Value Updates

**Standard groups (Info, Temps):** Reuse existing `handleRegisterValue()` -- values fill in via `data-register` attribute matching. Temperature color coding applied on value arrival using existing threshold logic (normal/elevated/critical).

**Cell voltage grid:** Each cell voltage arrives as a `register_value` message. The cell value element is identified by `data-register` attribute (format: `"Cell Voltages::cell_N"`). On arrival:
1. Update the cell's voltage text
2. Recalculate min/max/spread/avg from all arrived cell values
3. Update cell deviation color classes (`cell--good` / `cell--warn` / `cell--danger`)
4. Update summary row values (Min, Max, Spread, Avg)
5. Add `data-row-h__value--fresh` class to override container dimming

**Balance state:** The balance bitmap arrives as a single `register_value`. On arrival:
1. If bitmap is 0: show "Balanced" text with `balance-status--ok` class
2. If bitmap is non-zero: show "Balancing Active" with `balance-status--active` class + cell pills
3. Add `data-row-h__value--fresh` class to the balance status container

**Pack status:** Status registers (alarm, protection, fault, alarm2, protection2, fault2) arrive individually. The pack status card updates progressively:
1. Start in clear state (all-clear assumption)
2. As each status register arrives, if non-zero, switch to active state
3. Decoded fault strings update the fault list
4. Final state is determined after all status registers arrive (section_complete)

#### 2d. Section Complete

On `section_complete` for a pack read cycle:
1. Remove `content__body--refreshing` class from `content-body`
2. Trigger `triggerFlash('success')`
3. Update timestamp display
4. Resume auto-refresh cycle scheduling (if active)

#### 2e. Interaction with Phase 10 Features

| Feature | Pack Streaming Behavior |
|---------|------------------------|
| Refresh dimming (DISP-01) | Applied identically: container gets `--refreshing` class at cycle start, individual values get `--fresh` on arrival, container class removed on `section_complete` |
| Section caching (DISP-02) | Cache key: `bms:pack:{input}:{tower}:{pack}`. `updateCache()` called on each `register_value`. `restoreFromCache()` called after skeleton build. |
| Tooltips (DISP-03) | `data-register-addr` and `data-register-raw` attributes set on each value element by `handleRegisterValue()`. Tooltip hover behavior is generic and works on any element with these attributes. |

## Skeleton States

### Cell Grid Skeleton

```
+------------------------------------------+
| Cell Voltages                            |
|------------------------------------------|
| Min: --   Max: --   Spread: --   Avg: -- |
|                                          |
| [Cell 1] [Cell 2] [Cell 3] [Cell 4]     |
| [  --  ] [  --  ] [  --  ] [  --  ]     |
| [Cell 5] [Cell 6] [Cell 7] [Cell 8]     |
| [  --  ] [  --  ] [  --  ] [  --  ]     |
| ...                                      |
+------------------------------------------+
```

Each cell skeleton: `cell-voltage` div with `cell-number` span ("Cell N") and `cell-value` span (em-dash, `data-row-h__value--pending` class). No deviation color class applied until value arrives.

Summary skeleton: All 4 summary items (Min, Max, Spread, Avg) show em-dash as value. No spread color class applied until values arrive.

### Balance Skeleton

```
+------------------------------------------+
| Balance State                            |
|------------------------------------------|
| --                                       |
+------------------------------------------+
```

Single pending value element. On first `register_value` arrival, replaced with either "Balanced" or "Balancing Active" + pills.

### Pack Status Skeleton

```
+------------------------------------------+
| Pack Status                              |
|------------------------------------------|
| --                                       |
+------------------------------------------+
```

Starts as a neutral card. On status register arrivals, transitions to either `fault-card--clear` or `fault-card--active` styling.

## Interaction Contract

### Pack Selection Flow (unchanged user interaction, new data flow)

1. User clicks bitmap cell or changes pack selector
2. `sendSelectPack(input, tower, pack)` sends `select_pack` WebSocket message
3. `showPackLoading()` displays loading spinner
4. Backend sends `section_schema` with `pack_context` -- frontend builds skeleton
5. Backend streams `register_value` messages -- frontend fills values progressively
6. Backend sends `section_complete` -- frontend removes dim, shows flash

### Auto-Refresh on Pack View (unchanged)

1. On `section_complete`, if auto-refresh active, schedule next cycle after delay
2. Next cycle: send `read_cycle` with section `bms` -- backend detects pack mode and re-reads current pack
3. Before cycle starts: `applyRefreshDimming()` dims all existing pack values
4. Values stream in: each gets `--fresh` class to snap to full opacity
5. Cycle completes: container dim removed

### Pack Switch During Read (edge case)

1. User changes pack selector while current pack is being read
2. Frontend sends new `select_pack` message
3. Frontend shows loading spinner (clears current skeleton)
4. Backend cancels current read (if applicable) and starts new pack
5. New `section_schema` arrives -- fresh skeleton built
6. New values stream in
