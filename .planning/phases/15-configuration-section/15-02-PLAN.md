**Step 3: Modify `handleReadCycle()` to check readOnce**

Add a readOnce guard after the existing `reading.Load()` check:

```go
func (h *Hub) handleReadCycle(c *Client, msg InboundMessage) {
    if !h.connected { return }
    sec, ok := h.sections[msg.Section]
    if !ok { return }
    if sec.SubscriberCount() == 0 { return }
    if sec.reading.Load() {
        h.logger.Debug("skipping overlapping read_cycle", "section", msg.Section)
        return
    }
    // D-09: Skip re-read for cached read-once sections
    if sec.readOnce && sec.hasReadOnce {
        h.logger.Debug("skipping read_cycle for cached read-once section", "section", msg.Section)
        return
    }
    // BMS pack handling...
    h.triggerSectionRead(msg.Section)
}
```

**Step 4: Set hasReadOnce after successful read completion**

In the result dispatcher (the goroutine that reads from `h.results` channel and distributes to subscribers), detect when a `section_complete` message arrives for a readOnce section and set `hasReadOnce = true`.

Find where section_complete results are processed (in the `handleResult` or `run()` loop). After broadcasting section_complete for a readOnce section, set:

```go
if sec.readOnce && !sec.hasReadOnce {
    sec.hasReadOnce = true
```

This should be in the `handleSectionResult` or equivalent function in `hub.go` where `sectionResult` messages are processed and broadcast to subscribers.

**Step 5: Reset hasReadOnce on explicit refresh**

The `MsgTypeRefresh` handler already calls `h.triggerSectionRead(msg.Section)` directly, bypassing `handleReadCycle`. Before calling triggerSectionRead, reset the cache flag:

```go
case MsgTypeRefresh:

        sec.hasReadOnce = false
    }
    h.triggerSectionRead(msg.Section)
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-03 | Tampering | hub.go | mitigate | Configuration section is read-only. No WriteRegister calls in streamStandardRead path. The readOnce flag only suppresses reads -- it cannot enable writes. Verify no write path exists for configuration section. |
| T-15-04 | Denial of Service | section.go | accept | A malicious WebSocket client could spam read_cycle for configuration. The existing `reading.Load()` guard prevents overlapping reads, and readOnce prevents re-reads after cache. No new DoS surface added. |
| T-15-05 | Spoofing | hub.go | accept | Configuration values come from the inverter via Modbus TCP on trusted LAN. No authentication on Modbus is a protocol limitation, not introduced by this change. |
