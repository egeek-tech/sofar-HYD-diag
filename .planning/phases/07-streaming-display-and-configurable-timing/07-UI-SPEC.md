
**Behavior:**
- Inputs are always visible in the header bar (D-05).
- Values are direct numeric entry, no sliders (D-04).
- On change (blur or Enter key), send updated value via WebSocket `configure` message.
- Changed values take effect on next read cycle (D-06).
- Read Delay: min=100, max=5000, step=100, default=500.
- Pack Settle: min=500, max=10000, step=100, default=1000.
- Invalid values (out of range, non-numeric) are clamped to nearest valid value on blur.
- Values persist in `localStorage` key `sofar_timing` between sessions.

No new DOM elements -- these are CSS state classes applied to the existing `.data-row-h__value` elements.

**State: Unloaded (em dash placeholder)**
```css
.data-row-h__value--pending {
    color: var(--color-text-muted);         /* #888888 */
}
```
Content: the text `\u2014` (em dash character).

**State: Loaded (value arrived)**
```css
/* No extra class needed -- default .data-row-h__value styling applies */

```

**State: Stale (last known value after error)**
```css
.data-row-h__value--stale {
    color: var(--color-text-muted);         /* #888888 */
```

**Error icon for stale values (D-03):**

```css
    margin-left: var(--space-xs);           /* 4px */
}
```

### Modified Components

#### 1. Content Header Controls (`.content__header-controls`)

Add the timing controls container as a child, inserted before the auto-refresh button. Existing children (PV channel select, pack selectors, auto-refresh) remain unchanged.

**Updated insertion order:**
1. PV channel select (existing, shown for PV section)
2. Pack selector controls (existing, shown for BMS pack detail)
3. **Timing controls (new, always visible when connected)**
4. Auto-refresh button (existing)

**Visibility rule:** Timing controls container is shown (`display: flex`) when WebSocket connection is in `connected` state. Hidden (`display: none`) when disconnected.

#### 2. Data Row Rendering (`renderGroupCard` function)

Each `.data-row-h__value` element must receive a `data-register` attribute with the register address (hex string, e.g. `"0x0445"`) to enable targeted DOM updates when streaming messages arrive.

**Before (batch):** `body.textContent = ''` clears all, re-creates all DOM.
**After (streaming):** DOM is created once per section. Individual `.data-row-h__value[data-register="0xNNNN"]` elements are updated in-place via `querySelector`.

## Interaction Contract

### Streaming Update Flow

1. User navigates to a section (e.g. "System").
2. Frontend sends `subscribe` message to backend.
3. Backend begins reading registers sequentially with inter-read delay.
4. **Per-register:** Backend sends a streaming update message as each register is read.
5. Frontend finds the matching `[data-register]` element and replaces the em dash with the value.
6. After all registers in the section are read, the timestamp updates.
7. If auto-refresh is on, the cycle repeats.

### Loading Cycle Visual Sequence

| Step | Value Element State | CSS Class | Content |
|------|-------------------|-----------|---------|
| Section just activated | Pending | `.data-row-h__value--pending` | `\u2014` (em dash) |
| Streaming value arrives | Loaded | (none, default styling) | Register value text |
| Register read fails | Stale (if prior value exists) | `.data-row-h__value--stale` | Last known value + warning icon |
| Register read fails | Error (no prior value) | `.data-row-h__value--pending` | `\u2014` (em dash stays) |
| New cycle starts | Values stay visible | (none, default styling) | Previous values remain until replaced |

**Key behavior:** On a new read cycle, existing values are NOT cleared to em dashes. They remain visible until the new streaming value replaces them. This prevents a "flickering" effect between cycles.

### Timing Control Interaction

| Action | Behavior |
|--------|----------|
| User types a number | No immediate effect; value is staged |
| User presses Enter or blurs the input | Value is validated (clamped to min/max), sent via WebSocket `configure` message |
| Value is out of range | Clamped to nearest bound (100 or 5000 for read delay; 500 or 10000 for pack settle) |
| Value is non-numeric | Input reverts to previous valid value |
| Backend receives new timing | Applied on the next read cycle, no reconnection (D-06) |

### Timing Persistence

- On page load, read `localStorage.getItem('sofar_timing')`.
- Expected format: `{"readDelay": 500, "packSettle": 1000}`.
- If present, populate inputs with stored values and send `configure` on connect.
- On each timing change, write to localStorage.

## Accessibility Contract

| Concern | Implementation |
|---------|---------------|
| Timing input labels | `<label>` elements with `for` attribute matching input `id` |
| Read Delay input id | `timing-read-delay` |
| Pack Settle input id | `timing-pack-settle` |
| Input `aria-label` | Not needed (visible `<label>` provides accessible name) |
| Input `min`/`max` attributes | Set on the `<input>` element for native validation hints |
| Stale value state | Warning icon uses `\u26A0` character (screenreader-readable as "warning sign") |
| Pending value state | Em dash `\u2014` is read as "em dash" by screen readers -- acceptable for placeholder |
| Keyboard support | Tab order: Read Delay input -> Pack Settle input -> Auto-refresh button. Enter commits value. |
