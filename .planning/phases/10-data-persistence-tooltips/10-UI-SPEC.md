| Property | Value |

| Tool | none (vanilla CSS custom properties) |
| Preset | not applicable |
| Component library | none (vanilla HTML/JS) |
| Icon library | Unicode emoji (existing pattern) |
| Font | `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif` (existing `--font-family`) |

**Note:** This project is a Go backend with embedded vanilla HTML/JS/CSS. No build tools, no framework, no CSS preprocessor. All design tokens are CSS custom properties in `web/static/style.css`.

**Phase 10 is frontend-heavy.** Three visual features: (1) stale value dimming during refresh cycles (DISP-01), (2) section-level value caching with dimmed restoration (DISP-02), (3) parameter tooltips showing register metadata (DISP-03). ## Component Inventory (Phase 10 Scope)

### 1. Refresh Dimming (New Behavior -- DISP-01)

**Container-level dimming when a read cycle starts.**

| Property | Value |
|----------|-------|
| Trigger | Browser sends `read_cycle` message (or `subscribe` triggers immediate read) |
| Target | Content body container (`.content__body` or `.group-card` level) |
| CSS class | `.content__body--refreshing` applied to the content body container |
| Visual | All child value elements inherit `opacity: 0.5` via CSS |
| Transition | `opacity var(--refresh-dim-transition) ease` (150ms) |
| Removal | Class is NOT removed at section_complete. Instead, per-register un-dimming handles restoration. Container class removed at `section_complete` as a cleanup sweep. |

**Per-register un-dimming as values arrive:**

| Property | Value |
|----------|-------|
| Trigger | Each `register_value` message with no error |
| Target | Individual `.data-row-h__value` element matching `data-register` attribute |
| Behavior | Set `opacity: 1` directly on the element (overrides container-level dim) |
| Transition | `opacity var(--refresh-dim-transition) ease` (150ms) |
| Error case | If `register_value` has error, element stays at container opacity (dimmed) and gets `.data-row-h__value--stale` class with warning icon (existing Phase 9 behavior) |

**Coexistence with green flash (D-06):**

| Event | What Happens |
|-------|-------------|
| `read_cycle` sent | Container gets `.content__body--refreshing` (dims all values to 50%) |
| Each `register_value` | Individual value restored to full opacity |
| `section_complete` | Green flash fires (existing), container `.content__body--refreshing` removed (cleanup) |

**Distinction from error stale (D-03):**

| State | Class | Visual | Icon |
|-------|-------|--------|------|
| Never read | `.data-row-h__value--pending` | Em-dash, muted color | None |
| Refreshing (cycle in progress) | Container `.content__body--refreshing` | Previous value visible at 50% opacity | None |
| Error after all retries | `.data-row-h__value--stale` | Muted color + warning icon | Warning triangle (existing) |
| Fresh value | No special class | Full opacity, normal color | None |

Source: CONTEXT.md D-01 through D-07.

### 2. Section Value Cache (New Behavior -- DISP-02)

**In-memory JavaScript cache for section values, restored on navigation.**

| Property | Value |
|----------|-------|
| Data structure | Claude's discretion (Map or plain object) |
| Cache key | Section name (e.g., `"system"`, `"grid"`) for main sections. For pack drill-down: `"bms:pack:{input}:{tower}:{pack}"` |
| Cache value | Map of `data-register` key to `{ value: string, registerAddr: string, rawValue: string, timestamp: string }` |
| Update trigger | Every successful `register_value` message |
| Clear trigger | Disconnect event (D-12). Also cleared on page reload (in-memory only, no localStorage). |

**Navigation restoration flow:**

| Step | Action | Visual |
|------|--------|--------|
| 1. User navigates to a previously-visited section | `navigateToSection()` checks cache for this section | -- |
| 2. Cache exists | Schema is rendered. Cached values populate `data-register` elements. Container gets `.content__body--refreshing` class. | Values visible at 50% opacity |
| 3. No cache (first visit) | Schema renders with em-dash placeholders (existing behavior). No dimming class. | Em-dashes in muted color |
| 4. Subscribe sent | Fresh read cycle begins | Cached values dim, then progressively restore as fresh data arrives |

**Em-dash vs dimmed distinction (D-10):**

| Situation | Display | Meaning |
|-----------|---------|---------|
| First visit, no cache | Em-dash (`---`) in `--color-text-muted` | "Never read" |
| Return visit, cache exists | Previous value text at 50% opacity | "Read before, refreshing now" |
| After disconnect | Em-dash (`---`) in `--color-text-muted` | "No connection, no data" |

Source: CONTEXT.md D-08 through D-14.

### 3. Parameter Tooltip (New Component -- DISP-03)

**Custom dark tooltip on hover showing register metadata.**

| Property | Value |
|----------|-------|
| Trigger | Mouse hover on any `.data-row-h__value` element, 300ms delay before appearing |
| Dismiss | Mouse leaves the value element |
| Position | Centered above the value element. If insufficient space above, flip to below. |
| Max width | 200px |
| Z-index | `var(--tooltip-z-index)` (100) |

**Tooltip visual specification:**

| Property | Value |
|----------|-------|
| Background | `var(--tooltip-bg)` (#1a1a2e) |
| Text color | `var(--tooltip-text)` (#f0f0f0) |
| Font | `var(--tooltip-font)` (monospace stack) |
| Font size | 12px |
| Font weight | 400 |
| Line height | 1.4 |
| Padding | 8px 16px |
| Border radius | `var(--tooltip-radius)` (4px) |
| Arrow | 6px CSS triangle pointing down (same background color), centered horizontally |
| Shadow | `0 2px 8px rgba(0, 0, 0, 0.25)` |

**Tooltip content (three lines):**

```
Register: 0x0484
Raw: 3852
Last read: 16:03:42
```

| Line | Source | Format |
|------|--------|--------|
| Register address | `data-register-addr` attribute on value element | `0x` + 4-digit uppercase hex (e.g., `0x0484`) |
| Raw value | `data-register-raw` attribute on value element | Decimal integer as received from backend |
| Last read timestamp | `data-register-time` attribute on value element | `HH:MM:SS` (24-hour, from cache timestamp) |

**Tooltip for special states:**

| State | Content |
|-------|---------|
| Fresh value (normal) | All three lines shown |
| Error-stale (retries failed, previous value exists) | Register address + last known raw value + last read timestamp |
| Never read (em-dash, no prior value) | Register address only. Raw and timestamp lines omitted. |

**Implementation approach:** Claude's discretion per CONTEXT.md. Options: CSS-only `::before`/`::after` (limited to static content), or JS-managed tooltip element (required for dynamic content from data attributes). Given that content is dynamic, a single reusable JS-managed tooltip element positioned on hover is recommended.

Source: CONTEXT.md D-15 through D-19.

### 4. Disconnect Cache Clear (Modified Behavior)

| Property | Value |
|----------|-------|
| Trigger | `state_change:disconnected` WebSocket message |
| Actions | (1) Clear entire section value cache. (2) Reset all value elements to em-dash with `.data-row-h__value--pending` class. (3) Remove any `.content__body--refreshing` class. |
| Visual result | All sections show em-dash skeletons (D-14). Clean slate. |

Source: CONTEXT.md D-12, D-13, D-14.

## New CSS Classes

### `.content__body--refreshing`

```css
.content__body--refreshing .data-row-h__value {
    opacity: var(--refresh-dim-opacity, 0.5);
    transition: opacity var(--refresh-dim-transition, 150ms) ease;
}
```

Applied to `.content__body` container when a read cycle begins. Dims all value elements to 50%. Individual values override this with inline `opacity: 1` as they arrive.

### `.data-row-h__value--fresh`

```css
.data-row-h__value--fresh {
    opacity: 1 !important;
    transition: opacity var(--refresh-dim-transition, 150ms) ease;
}
```

Applied to individual value elements as each `register_value` message arrives during a refresh cycle. Overrides the container-level dim. Removed when the next cycle starts (container dim reapplied).

**Alternative:** Use inline `style="opacity:1"` instead of a class. Either approach works; the executor may choose based on implementation simplicity.

### `.param-tooltip`

```css
.param-tooltip {
    position: absolute;
    background: var(--tooltip-bg, #1a1a2e);
    color: var(--tooltip-text, #f0f0f0);
    font-family: var(--tooltip-font);
    font-size: 12px;
    font-weight: 400;
    line-height: 1.4;

    border-radius: var(--tooltip-radius, 4px);
    box-shadow: 0 2px 8px rgba(0, 0, 0, 0.25);
    z-index: var(--tooltip-z-index, 100);
    pointer-events: none;
    white-space: nowrap;

```

Single reusable tooltip element, positioned absolutely via JavaScript on hover. Arrow points down (tooltip above value).

## Data Attribute Contract

Each `.data-row-h__value` element requires these data attributes for tooltip support:

| Attribute | Format | Set By | Example |
|-----------|--------|--------|---------|
| `data-register` | `{group}::{name}` | Schema render (existing) | `"System Info::Inverter SN"` |
| `data-register-addr` | `0x` + 4-digit hex | `handleRegisterValue()` from backend message | `"0x0445"` |
| `data-register-raw` | Decimal string | `handleRegisterValue()` from backend message | `"3852"` |
| `data-register-time` | `HH:MM:SS` | `handleRegisterValue()` from `Date.now()` | `"16:03:42"` |

**Backend message extension:** `RegisterValueMessage` in `internal/hub/message.go` must include two new fields:

| Field | Type | JSON key | Description |
|-------|------|----------|-------------|
| RegisterAddr | `uint16` | `register_addr` | Modbus register address (e.g., 0x0445) |
| RawValue | `string` | `raw_value` | Raw register value before scaling/formatting (e.g., "3852") |

The frontend formats `register_addr` as hex (`"0x" + addr.toString(16).toUpperCase().padStart(4, '0')`).

## Interaction States

### Read Cycle Dimming Flow

```
[User on section] ---> [read_cycle sent] ---> [All values dim to 50%]
                                                      |
                                               [register_value #1] ---> [Value #1 snaps to 100%]
                                               [register_value #2] ---> [Value #2 snaps to 100%]
                                               ...
                                               [section_complete] ---> [Green flash + cleanup]
```

### Section Navigation with Cache

```
[User navigates to System] ---> [Cache empty?]
       |                              |
       |                         YES: Show em-dash skeletons
       |                         NO:  Populate from cache at 50% opacity
       |
       +---> [Subscribe sent] ---> [Read cycle begins]
                                         |
                                   [Values progressively restore to 100%]
```

### Tooltip Interaction

```
[Mouse enters value element] ---> [300ms delay] ---> [Tooltip appears above]
[Mouse leaves value element] ---> [Tooltip disappears immediately]
[Mouse enters value, leaves before 300ms] ---> [No tooltip shown]
```

### Disconnect Reset

```
[Disconnect received] ---> [Cache cleared]
                      ---> [All values reset to em-dash]
                      ---> [Refreshing class removed]
                      ---> [Tooltip hidden if visible]
```

## Accessibility Notes

- **Tooltip content:** Tooltip data (register address, raw value, timestamp) is presentational/diagnostic metadata, not essential for screen reader users. The tooltip element should have `role="tooltip"` and the value element should have `aria-describedby` pointing to the tooltip's `id` when visible.
- **Opacity dimming:** 50% opacity maintains sufficient contrast for reading (14px body text at 50% opacity on white card background still meets WCAG AA for large text). The dim state is transient (values restore within seconds as reads complete).
- **No keyboard tooltip trigger:** Tooltips are hover-only (D-17). This is acceptable for a desktop diagnostic tool where the metadata is supplementary. The register addresses are available programmatically via data attributes for any assistive technology that needs them.
